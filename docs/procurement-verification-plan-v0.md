# 조달 법령 검증 계획 v0

기준일: 2026-07-12

목표: `오류 가능성이 없도록 철저히 검증된 법령 제공`. 기존 korea100 검증 파이프라인(`sync:sources`, `verify:articles`, `check:freshness`)을 승계하되, 조달 도메인의 규범 구조에 맞게 정책을 강화한다.

## 1. 검증 대상 규범의 층위

| 층위 | 규범 | 국가법령정보센터 연결 방식 | 비고 |
| --- | --- | --- | --- |
| 1 | 법률 (국가계약법, 지방계약법, 조달사업법, 전자조달법, 판로지원법 등) | 법령 ID + MST | 기존 파이프라인 그대로 |
| 2 | 시행령·시행규칙 | 법령 ID + MST | 기존 파이프라인 그대로 |
| 3 | 기획재정부 계약예규 | 행정규칙 ID + 일련번호 | 조달 콘텐츠의 실질 중심. 개정 잦음 |
| 4 | 행정안전부 예규 (지방계약 기준) | 행정규칙 ID + 일련번호 | 지방계약 제도에 필수 |
| 5 | 조달청 고시·훈령·지침 | 행정규칙 ID + 일련번호 | MAS, 우수조달물품 등 조달청 고유 제도의 근거 |
| 6 | 조달청 공개 매뉴얼·나라장터 안내 | 법령정보센터 밖 | `unresolved` 사유 코드 또는 `fieldVerification`로 관리 |

korea100 대비 가장 큰 차이: 층위 3~5(행정규칙)가 콘텐츠의 중심이 된다. 행정규칙 전문 대조는 기존 파이프라인이 지원하므로 기술 부채는 없고, 아래 정책 부채를 채운다.

## 2. 검증 상태 정의 (승계 + 조달 보강)

- `article-verified`: 캔버스와 모든 절차 노드의 명시 조문이 현행 원문과 대조됨. 공개 기본 기준.
- `source-linked`: 규범 원문은 연결됐으나 일부 조문·범위 확인이 남음. 남은 사유를 `unresolved`에 코드로 기록한 경우에만 공개 가능.
- `needs-review`: 공개 불가.

조달 버전 추가 규칙:

1. 금액·비율·기한 수치가 있는 제도는 해당 수치의 근거 조문이 대조를 통과해야만 `article-verified`를 부여한다. 수치 근거가 미검증이면 수치를 비우고 `source-linked`로 공개한다.
2. 통칭 사용 금지: `적격심사기준` 같은 통칭은 본문 표기용이고, 데이터에는 공식 제명(`officialName`)과 행정규칙 ID를 반드시 기록한다.
3. 국가·지방 병기 제도는 두 계열의 근거를 각각 연결한다. 한쪽만 연결된 상태로 `공통`이라 표기하지 않는다.

## 3. 검증 워크플로

### 도구 방침

검증 도구는 korea100 파이프라인(`sync:sources`, `verify:articles`, `check:freshness`, `validate:data`)을 수정 없이 그대로 사용한다. 조달 버전에서 새로 만드는 것은 도구가 아니라 작성 규칙이다.

### 제작 시 (제도 1건 기준) — 원문 우선

초안을 쓰고 나서 검증하는 순서가 아니라, 원문을 먼저 확보하고 원문에서 초안을 쓴다. 조문 존재 검증은 존재하는 조문에 붙은 틀린 설명을 잡지 못하므로, 환각 차단은 작성 단계에서 한다.

1. 원문 확보: 근거 법령·행정규칙을 국가법령정보센터에서 먼저 찾아 원문을 확보하고 `sync:sources`로 연결한다.
2. 초안 작성: 확보한 원문을 옆에 두고 인용하며 쓴다. 기억·블로그·해설서만으로 절차나 요건을 서술하지 않는다. 조문번호는 대조 통과 전까지 `검증 전` 후보로 표기한다.
3. `LAW_OC=... npm run verify:articles -- --write`로 명시 조문을 현행 원문과 대조한다.
4. 불일치 항목은 초안을 고친다. 원문에 없는 조문을 각주로 남기는 방식은 쓰지 않는다.
5. 수치 항목(`thresholds`)은 대조 결과와 별도로 사람 눈으로 원문 재확인 후 기준일을 기록한다. 부칙 시행일과 경과조치를 함께 확인한다.
6. `npm run validate:data` 통과 후 공개 빌드에 포함한다.

### 선택 백로그 (파일럿 후 필요성 재평가)

파일럿 10개 운영에서 오류가 실제로 새는 지점이 확인되면 도입을 검토한다. v0에서는 도입하지 않는다.

- 스니펫 문자열 대조: 노드·수치에 원문 인용 스니펫을 저장하고 검증기가 원문 존재 여부를 자동 대조
- 빌드 게이트 강화: 미검증 조문 인용·근거 없는 수치가 있으면 `validate:data`가 빌드를 실패시키는 규칙
- 실무자 검독: 조달 실무 경험자 1인 이상의 검독을 완료 기준에 포함

### 개정 감시 (운영)

- GitHub Actions 주간 freshness 점검을 승계하되, 계약예규·조달청 고시를 감시 대상 레지스트리에 반드시 등록한다.
- 변경 감지 시 이슈를 생성하고, 영향받는 제도의 `asOfDate`를 갱신할 때까지 해당 제도 상단에 `개정 확인 중` 표시를 검토한다(웹앱 기능 백로그).
- 계약예규는 연 단위 이상으로 개정될 수 있으므로, 파일럿 10개 공개 시점에 감시 레지스트리 커버리지를 검수 항목으로 넣는다.

### 현장 검증 큐

법령으로 확인할 수 없는 항목은 `fieldVerification`으로 분리해 공개 검증 대장에 올린다. 조달 버전에서 예상되는 대표 항목:

- 나라장터 화면 흐름과 실제 클릭 순서
- 심사 실무 소요 기간 (법정 기한과 실제 처리 기간의 차이)
- 수요기관별 내부 검토 단계 (일상감사, 계약심사 운영 편차)
- 보완 요구·반려의 실무 빈도와 사유

제보 수집 시 개인 정보와 비공개 내부 자료를 받지 않는다는 고지를 유지한다.

## 4. 조달 특유의 오류 위험과 통제

| 위험 | 예 | 통제 |
| --- | --- | --- |
| 수치 개정 누락 | 수의계약 한도, 낙찰하한율 변경 | 수치=값+근거+기준일 세트, 주간 감시, 부칙 확인 |
| 국가/지방 근거 혼용 | 국가계약법 조문을 지방계약 설명에 인용 | `scope` 필드와 계열별 출처 분리 연결 |
| 예규 조문 이동 | 예규 전부개정으로 조문 번호 재배열 | 행정규칙 일련번호 고정 + 재대조, 통칭 의존 금지 |
| 상하위 규범 혼동 | 법 조문과 예규 조문을 같은 층위로 인용 | `kind`/`sourceType`으로 층위 표기, 표기 규칙 검증기 유지 |
| 시스템 절차의 법령 오인 | 나라장터 화면 단계를 법정 절차로 서술 | 시스템 단계는 `fieldVerification` 분류 강제 |
| 시행 예정 개정 | 공포됐으나 미시행인 개정 내용 게시 | `verification.notes`에 시행 예정 사항 기록, 현행 기준 서술 |

## 5. 공개 문구

모든 상세 페이지에 다음을 표시한다.

- 법령 기준일(`asOfDate`)
- 비자문 고지: `이 자료는 공공조달 제도 이해를 위한 참고 자료이며, 개별 입찰·계약 사건에 대한 법률 자문이나 조달청·기획재정부·행정안전부의 공식 해석을 대신하지 않습니다. 실제 입찰·계약 전에는 반드시 해당 공고문과 현행 법령 원문을 확인하십시오.`
- 검증 상태 배지와 그 의미 링크

## 6. 검증 요약 보고

파일럿 10개 공개 시점에 `docs/verification-summary.md` 형식을 승계한 조달 버전 요약을 발행한다. 포함 항목:

- 제도 수, 절차 노드 수, 공식 원문 연결 수
- 층위별(법률/령·규칙/예규/고시) 연결 현황 — 조달 버전 신규 지표
- 명시 조문 인용 수와 불일치 0건 확인
- 수치 항목(`thresholds`) 검증 현황 — 조달 버전 신규 지표
- 미해결 사유 코드별 집계와 한계 서술

한계 문단은 승계한다: 조문 존재 확인은 인용의 법적 의미 완전성, 개별 사건 적용, 판례·유권해석 일치를 보장하지 않는다.

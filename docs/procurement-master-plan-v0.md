# 그 많던 조달은 어떻게 했을까 — 마스터플랜 v0

기준일: 2026-07-12

## 1. 프로젝트 정의

- 제목: 그 많던 조달은 어떻게 했을까
- 부제: 한 장으로 끝내는 조달제도 100

`한 장으로 끝내는 대한민국 제도 100`의 조달청 버전을 만든다. 대한민국 공공조달·계약 제도를 한 장의 구조도로 읽는 공개 웹서비스다. 브랜드 제목은 기억과 공유를 맡고, 부제가 검색 키워드(조달제도)와 형식 약속(한 장, 100)을 맡는다. 서비스 화면과 공유 메타데이터에는 제목과 부제를 항상 함께 쓴다.

`100`은 상한이 아니라 큐레이션 약속이다. 100개를 넘어서는 확장은 `핵심 100선 + 심화` 구조로 수용하며 이름은 유지한다. 100개 이상에서의 탐색은 이름이 아니라 검색·분야 필터·이용자 태그가 해결한다.

> 기업에는 비즈니스 모델이 있듯이, 공공조달에는 계약 모델이 있다.

### 목적

1. 조달시장에 처음 진입하는 업체가 `무엇을, 어떤 순서로, 어떤 근거로` 해야 하는지 한 장으로 파악한다.
2. 조달요청을 해야 하는 수요기관 담당자가 자기 기관의 다음 행동을 찾는다.
3. 계약업무를 처음 맡은 공무원이 계약방법 결정부터 사후관리까지 전체 지도를 갖는다.
4. 모든 법령 인용은 국가법령정보센터 현행 원문과 대조한 뒤에만 공개한다. 검증 전 내용은 검증 전임을 표시한다.

### 이 프로젝트가 아닌 것

- 법률 자문, 유권해석, 개별 입찰·계약 사건에 대한 판단
- 실시간 입찰 정보나 나라장터 공고 중계
- 특정 업체의 낙찰을 돕는 컨설팅

## 2. 핵심 이용자 3인

| 이용자 | 상황 | 필요한 것 |
| --- | --- | --- |
| 신규 조달업체 | 조달시장 진입을 준비하거나 첫 입찰을 앞둠 | 등록→입찰→계약→대금까지의 순서, 자격 요건, 마감·기한, 흔한 탈락 사유 |
| 수요기관 담당자 | 물품·용역·공사가 필요해 조달요청을 해야 함 | 중앙조달 대상 여부, 조달요청 절차, 규격 작성, 검사·검수와 대가 지급 |
| 계약담당 공무원 | 인사이동으로 계약업무를 처음 맡음 | 계약방법 결정 트리, 낙찰자 결정 방식, 계약금액 조정, 제재와 분쟁 대응 |

세 이용자는 같은 제도를 서로 다른 lane에서 본다. swimlane 구조가 이 요구에 정확히 맞는다. 모든 제도 페이지는 `이 장을 읽는 사람` 태그(업체/수요기관/계약공무원)를 갖고, 홈에서 이용자 유형으로 필터할 수 있어야 한다.

## 3. 조달 도메인의 법령 위계

콘텐츠와 검증 정책의 기반이 되는 규범 구조. 명칭은 후보이며 공개 전 국가법령정보센터 공식 제명과 대조한다.

1. 법률: 국가를 당사자로 하는 계약에 관한 법률(국가계약법), 지방자치단체를 당사자로 하는 계약에 관한 법률(지방계약법), 조달사업에 관한 법률(조달사업법), 전자조달의 이용 및 촉진에 관한 법률, 중소기업제품 구매촉진 및 판로지원에 관한 법률(판로지원법) 등
2. 대통령령·부령: 각 법의 시행령·시행규칙
3. 계약예규(기획재정부, 행정규칙): 정부 입찰·계약 집행기준, 예정가격 작성기준, 적격심사기준, 공사·물품·용역 계약일반조건, 협상에 의한 계약체결기준 등
4. 행정안전부 예규: 지방자치단체 입찰 및 계약 집행기준 등 지방계약 하위 기준
5. 조달청 고시·훈령·지침: 다수공급자계약 업무처리규정, 우수조달물품 지정·관리 규정, 입찰참가자격 등록규정 등

조달 제도의 실무 규칙 대부분은 3~5층(행정규칙)에 있다. 기존 파이프라인이 행정규칙 ID·일련번호 연결과 전문 대조를 이미 지원하므로 그대로 확장한다. 상세 정책은 `procurement-verification-plan-v0.md`.

주의할 구조적 특성 두 가지.

- 국가계약과 지방계약의 이원 체계: 같은 이름의 제도(수의계약, 적격심사 등)라도 국가·지방의 근거와 기준이 다르다. 각 제도는 적용 범위(국가/지방/공통)를 명시한다.
- 금액·비율 기준의 민감성: 수의계약 한도, 낙찰하한율, 보증금률 같은 수치는 개정이 잦고 오류 시 실무 피해가 크다. 수치는 반드시 조문 대조를 통과한 것만 싣고, 노드 단위로 기준일을 관리한다.

## 4. 재사용 전략

이 저장소는 korea100의 전체 인프라를 물려받았다. 다시 만들지 않는 것과 새로 만드는 것을 구분한다.

### 그대로 재사용

- 데이터 계약: `Institution` 스키마(9칸 캔버스 + 상태 인식형 프로세스), `docs/data-contract.md`
- 검증 파이프라인: `sync:sources`, `verify:articles`, `check:freshness`(LAW_OC), 조문 파서와 테스트
- 웹앱: Next.js 정적 내보내기, 검색·필터·비교, 상세 swimlane, 검증 대장, 요청 폼
- 품질 게이트: `validate:data`, lint, audit, 빌드
- 디자인 시스템: `DESIGN.md` (조달 브랜드 컬러 여부는 2단계에서 결정)

### 조달 버전으로 신규 작성

- 제도 후보 100개 topic map → `procurement-topic-map-v0.md`
- 조달 특화 한 장 템플릿(이용자 관점, 금액 기준, 국가/지방 구분) → `procurement-one-page-template-v0.md`
- 법령 검증 계획(계약예규·고시 중심) → `procurement-verification-plan-v0.md`
- manifest, 제도 JSON 100개, 현장 검증 큐
- README·서비스 문구의 조달 리브랜딩

### 결정 필요 (2단계 진입 전)

- 기존 101개 제도 데이터의 처리: 삭제, 별도 보존 브랜치, 또는 `공공조달·국가계약` 1건만 승계 후 분해. 권고: 별도 브랜치에 보존하고 main은 조달 전용으로 비운다.
- 이용자 유형 필터를 위한 스키마 확장(`audience` 필드): 데이터 계약 v3로 반영

## 5. 단계별 로드맵

### 0단계 — 기획 확정 (지금)

- 마스터플랜, topic map, 템플릿, 검증 계획 문서 확정
- 파일럿 10개 선정과 순서 합의

산출물: 본 문서 포함 계획 문서 4종.

### 1단계 — 파일럿 1개 완주 (1~2주)

- 파일럿 1호 `조달업체 등록(입찰참가자격 등록)`을 데이터 계약 전체 깊이로 제작
- 근거 법령·행정규칙을 API로 연결하고 조문 대조까지 통과시켜 `article-verified` 달성
- 이 과정에서 조달 도메인 특유의 스키마 부족분(이용자 태그, 금액 기준, 국가/지방 구분)을 발견해 데이터 계약 v3 초안 작성

완료 기준: 파일럿 1건이 로컬 웹앱에서 캔버스+구조도로 렌더링되고 `validate:data`를 통과한다.

### 2단계 — 파일럿 10개 공개 (4~6주)

- 이용자 3인을 모두 커버하는 10개 제작 (선정안은 topic map 참조)
- 저장소 리브랜딩: README, 홈 문구, manifest 교체
- 검증 대장 공개, 비자문 고지 문구를 조달 맥락으로 수정
- 첫 공개 후 제작 요청 폼으로 다음 수요 수집

완료 기준: 10개 모두 `article-verified` 또는 사유가 기록된 `source-linked`, 빌드·검증 전체 통과, 공개 URL 배포.

### 3단계 — 100개 확장 (분기 단위)

- 분류별 10개 단위로 확장, 매 배치마다 `verify:articles` 재실행
- 계약예규 개정 감시를 주간 freshness 체크에 포함
- 이용자 유형별 진입 페이지(업체 처음 오신 분 / 수요기관 / 계약담당) 추가

### 4단계 — 심화 (선택)

- 제도 간 여정 연결: `등록 → 입찰 → 계약 → 이행 → 대금`을 잇는 여정 뷰
- 금액 기준 조견표(수의계약 한도, 보증금률 등) 자동 생성 페이지
- 국가 vs 지방 계약 비교 뷰
- 법제처 확장 데이터(법령해석례·판례·행정심판례·위원회 결정문) 연계 — 계획은 `procurement-lawdata-extension-v0.md`, 시범 적용은 2단계 파일럿 제작 중 2~3개 제도에서 시작

## 6. 오류 방지 원칙 (요약)

1. 원문 우선 작성: 근거 원문을 먼저 확보하고 원문에서 초안을 쓴다. 조문 번호는 기억이나 2차 자료로 쓰지 않으며, 국가법령정보센터 현행 원문 대조를 통과한 것만 공개한다. 검증 도구는 기존 파이프라인을 그대로 사용한다.
2. 금액·비율·기한 수치는 근거 조문과 기준일을 함께 표기한다.
3. 법령상 절차와 실무 관행 추정은 상태로 구분한다(`fieldVerification`, `confidence`).
4. 계약예규·고시처럼 개정이 잦은 규범은 주간 변경 감시 대상에 반드시 포함한다.
5. 모든 페이지에 기준일과 비자문 고지를 표시한다. 문구 예: `이 자료는 제도 이해를 위한 참고 자료이며, 개별 입찰·계약에 대한 법률 자문이나 조달청·기획재정부의 공식 해석을 대신하지 않습니다.`
6. 검증 요약(`verification-summary`)의 한계 문단을 조달 버전에도 유지한다: 조문 존재 확인이 법적 의미의 완전성을 보장하지 않는다.

## 7. 리스크와 대응

| 리스크 | 내용 | 대응 |
| --- | --- | --- |
| 예규 개정 빈도 | 계약예규·고시는 법률보다 개정 주기가 짧다 | freshness 주간 점검에 행정규칙 포함, 노드별 기준일 관리 |
| 금액 기준 오류 | 한도·요율 수치 오류는 실무 피해 직결 | 수치는 조문 대조 필수, 미검증 수치 게시 금지 |
| 국가/지방 혼동 | 유사 제도의 근거·기준 차이 | 제도별 적용 범위 필드, 필요 시 국가·지방 별건 분리 |
| 내부지침 의존 | 조달청 내부 매뉴얼은 법령정보센터 밖에 있을 수 있음 | `unresolved` 사유 코드로 기록, 공개 문서만 인용 |
| 범위 과대 | 방위사업(방위사업법)까지 넓히면 검증 부담 급증 | v0 범위에서 방위사업 조달은 제외, 관련 제도로만 언급 |
| 기존 101개와의 혼선 | 리브랜딩 전 혼재 기간 | 2단계에서 일괄 교체, 이전 데이터는 브랜치 보존 |

## 8. 성공 지표

- 파일럿 10개의 `article-verified` 비율
- 이용자 유형 필터 사용률과 유형별 상세 진입

- 조달 실무 커뮤니티·수요기관에서의 재방문

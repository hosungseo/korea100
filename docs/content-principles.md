# 콘텐츠 작성 4대 원칙 (운영자 지시, 2026-07-12)

계약업무의 특성상 아래 네 가지에 따라 **가능한 계약과 절차가 완전히 달라진다**. 제도 콘텐츠를 작성·수정·검증할 때 이 네 가지를 항상 명시하는 것을 최우선 품질 기준으로 삼는다.

| # | 요소 | 콘텐츠에 반영하는 위치 | 예 |
| --- | --- | --- | --- |
| 1 | **누가** (담당자·행위주체) | `submittedDocuments[].actor`, process `lanes`/`actor`, `stakeholders` | 조달업체 / 수요기관 담당자 / 계약담당공무원 / 조달청 |
| 2 | **어떤 서류를** | `submittedDocuments[].documents`, process `output_documents` | 물품구매요청서, 견적서, 이행실적증명, 검사조서 |
| 3 | **언제까지** (법정 기한) | process `deadline`, `applicability`·절차 서술의 기한 문구 | 청구일부터 14일 이내(선금), 5일 이내(대가), 만료 20일 전(갱신) |
| 4 | **어떤 사업종류·금액구간** (적용 대상) | `canvas.applicability` | 종합공사 추정가격 30억원 이상 조달요청 의무, 100억원 미만 적격심사 |

작성 규칙:

1. 새 제도 JSON은 위 4요소가 비어 있으면 완성으로 보지 않는다. 기한과 금액 구간은 근거 조문·기준일과 함께 적는다.
2. 4요소 중 법령 원문으로 확인하지 못한 것은 추측으로 채우지 말고 `fieldVerification`에 올린다.
3. 검증자는 4요소 누락·불일치를 일반 조문 오류와 같은 등급(must-fix)으로 다룬다.
4. 캔버스 구성(2026-07-12 개편): `절차와 법적 근거` / `적용 대상`(구 돈의 흐름) / `제출서류`(담당자별, 구 문서의 흐름) / `병목` / `관련 제도` / `현장 검증 필요`. 목적은 상세 페이지 제목 아래 설명으로 통합했고, 개선점·권한 관계 블록은 화면에서 제거했다(권한 데이터는 비교 화면 `핵심 기관`에서 사용).
5. 분류 체계는 조달 업무의 순차 응집도를 따른다: 등록·자격 → 지정·판로 → 발주 → 공고 → 입찰 → 심사·평가 → 계약 체결 → 이행·대금 → 분쟁·권리구제 → 사후관리·제재 (A안, 2026-07-13; 전체 배치는 docs/taxonomy-order.json).
6. 콘텐츠 정렬(priority): 1차 정렬은 절차의 순서(위 분류 순), 2차 정렬은 같은 분류 안에서 **일반적 내용 → 특수한 내용** 순. 예: 발주에서 계약방법 결정(모든 계약에 일반) → 조달요청(중앙조달 특수), 계약·이행에서 선금·대가(일반) → 물가변동 조정(조건부) → 다수공급자계약(조달청 특수).

## 법령 확인·출처 우선순위 규칙 (운영자 지시, 2026-07-12 추가)

7. **법령정보는 반드시 법제처 국가법령정보센터(law.go.kr)를 통해서만 확인한다.** 웹 검색 스니펫·블로그·해설서는 후보 탐색용일 뿐 최종 근거가 될 수 없다. 이 리포에서 법제처 확인 수단은 두 가지다: ① Open API 기계 대조(`law-verify` Actions 러너, `verify:articles`/`sync:sources`), ② API로 받아온 원문 사본(`sources/law-cache/`) 직접 대조. 둘 다 불가능한 세션에서 작성한 내용은 반드시 `needs-review`로 표시하고 법제처 확인 전까지 승격하지 않는다.
8. **조달청이 발표한 규정(행정규칙)을 우선시한다.** 같은 주제에 국가계약법령(기획재정부 계약예규)과 조달청 고시·훈령·지침이 함께 존재하면, 조달 실무 서술·수치·절차는 조달청 규정을 우선 인용하고 상위 법령은 근거 사슬로 연결한다.
9. **조달청 행정규칙 확인 절차**: 제도마다 (a) 조달청 발표 행정규칙이 존재하는지 먼저 확인하고, (b) 법제처 국가법령정보센터에서 그 행정규칙이 조회되는지 확인한다(`search_admin_rule` → `get_admin_rule`). (c) 법제처에서 조회되지 않는 규칙·기준은 조달청 공식 홈페이지(pps.go.kr)의 법령정보/행정규칙 게시를 출처로 삼되, `verification.unresolved`에 `external-official-document` 사유로 기록하고 officialUrl 대신 워크시트에 출처 URL을 남긴다.
10. **행정규칙(계약예규·조달청 고시 등)을 인용할 때는 반드시 현행(최신) 여부를 확인한다.** 법제처 검색 결과의 `[현행]` 표시와 발령일·시행일을 확인해 기록하고, 연혁(과거) 버전 인용을 금지한다. `sync:sources`가 연결하는 일련번호는 현행 기준이며, 주간 freshness 점검으로 개정을 감시한다. 검증자는 인용된 행정규칙의 발령번호·발령일이 최신인지 원문 캐시 헤더와 대조한다.
11. **법적 근거 나열 순서**: 규범 층위 순서(법률 → 대통령령 → 총리령·부령 → 행정규칙·고시·지침)로 적는다. 각 층위에는 여러 근거가 올 수 있으며, **같은 층위 안에서는 일반 규정(기획재정부 계약예규 등)을 먼저, 조달청 규정을 다음 행에** 적는다. `canvas.legalBasis` 배열 순서가 화면 표시 순서다.

## 표현·검증 표시 규칙 (운영자 지시, 2026-07-14 추가)

12. **조달청 행정규칙 우선 전면 적용(강화)**: 규칙 8을 실무 서술·수치·절차·흐름도(process 노드/게이트/조건)에 예외 없이 적용한다. 같은 주제에 조달청 고시·훈령·지침이 있으면, 흐름의 기준 금액·기한·판정 조건·담당 행위자를 **조달청 규정 기준으로 재구성**하고 상위 법령(법률·시행령·계약예규)은 근거 사슬로만 연결한다. 기존 콘텐츠도 조달청 규정 우선 관점으로 재검토해 흐름도가 바뀌면 반영한다. 법령 간 관계(상위법 위임 → 하위 규칙 구체화, 일반법 ↔ 특별법, 준용·배제)를 원문으로 확인해 근거 사슬에 반영한다.
13. **사용자 노출 텍스트에 내부 도구·변수·프로세스 용어를 절대 쓰지 않는다.** `LAW_OC`, `law-cache`, `WebSearch`, `sync:sources`, `verify:articles`, CLI 플래그, 파일 경로, 검증 파이프라인 단계명 등은 화면에 표시되는 어떤 필드(process 노드/게이트/legal_basis text, canvas, oneLiner, purpose 등)에도, 그리고 사람이 읽을 수 있는 `verification.notes`·`scope`에도 넣지 않는다. 검증 상태는 "법제처 국가법령정보센터 현행 원문으로 확인" 같은 사용자 언어로만 표현한다. 미확인 사항은 내부 용어 없이 `fieldVerification`에 사용자 언어로 남긴다.
14. **조문확인(검증 마크) 상호작용**: 흐름도 노드의 조문확인 마크를 누르면 **인용 조문 원문이 담긴 팝업**이 뜨고, 팝업 안에는 각 근거별 **[근거법령 바로가기]**(에메랄드색) 버튼으로 법제처 현행 원문으로 이동한다. 팝업에는 조문의 **현행 시행일**과 **원문 확인일**도 표시한다(캔버스 `법적 근거` 블록의 법령 링크는 기존대로 팝업 없이 바로 이동). 조문확인 마크가 팝업 없이 곧바로 외부로 튕기지 않게 한다.

15. **조문확인 팝업은 반드시 조문 '원문'만 담는다(운영자 지시, 2026-07-14).** 편집·요약본을 조문 내용으로 보여주지 않는다. 사용자 편의를 위한 요약이 환각(원문과 다른 서술)을 낳을 위험보다, **국가법령정보센터 현행 원문을 그대로 제공**하는 것을 우선한다. 구현: 원문은 `verification.articleTexts`(법제처 Open API 현행 원문 추출)에서만 가져와 표시하고, 원문이 없으면 요약으로 대체하지 말고 "원문 미수록 — 근거법령 바로가기로 확인" 안내만 노출한다. 원문 추출 시 **인용 조문 본문에 다른 조문 내용이 섞여 들어가지 않도록**(행 시작의 다른 `제N조` 헤더에서 절단) 보장한다. `scripts/populate-article-texts.mjs`가 이 규칙을 구현하며, 문장 중간의 상호참조(예: "법 제7조제1항")는 절단 대상이 아니다.

## 환각 방지·법령정보 MCP 철칙 (운영자 지시, 2026-07-15 규범화)

16. **환각 방지 프로토콜 (korean-law-mcp 기법 이식, `docs/anti-hallucination-review.md`)** — 작성·검증 에이전트 공통 철칙:
    - (a) **실패 마커는 실패다.** 법령 도구 출력의 `[NOT_FOUND]`·`[NO_CITATIONS_FOUND]`·`[HALLUCINATION_DETECTED]`·`[ERROR]`는 데이터 없음/오류 신호다. 추측으로 메꾸지 말고 도구가 제안한 재시도 질의로만 재시도하거나 실패를 그대로 보고한다. "인용 없음"은 "검증 성공"이 아니다.
    - (b) **제출 전 셀프체크.** 초안의 조문 인용을 MCP `legal_analysis(mode=verify_citations)` 또는 원문 재조회로 스스로 교차검증한 뒤 제출한다.
    - (c) **요지는 원문 밀착.** `legal_basis.text`(요지)는 원문에 실재하는 표현·수치로만 쓴다. 통합 시 기계 내용 대조(`verify:citation-content`, bigram containment ≥ 0.5)가 돌므로 원문에 없는 서술은 요지가 아니라 `fieldVerification`에 넣는다.
    - (d) **결정적 3층 게이트가 최종 판정한다.** 조문 실존 대조(`verify:articles`) + 내용 대조(`verify:citation-content`) + 전수조사(`audit-legal-bases`) — 모두 비-LLM. LLM 검토는 이 게이트를 대체하지 못한다.
    - (e) **잘못된 원문보다 미수록.** 원문 추출이 불확실하면(별표 혼입, 절단 등) 해당 `articleTexts` 항목을 넣지 않는다 — 팝업은 "원문 미수록 + 근거법령 바로가기" 안내로 폴백한다.

17. **법령정보 MCP 사용 규칙** — 루트 `.mcp.json`에 `korean-law` 서버가 등록되어 있다:
    - (a) **원문 근거 작성.** 법 조문·수치·기한을 쓰기 전에 MCP 도구(`search_law`·`get_law_text`·`get_batch_articles`·`search_admin_rule`·`get_admin_rule`)로 법제처 현행 원문을 먼저 조회해 읽고, 그 원문에 근거해 작성한다. WebSearch는 제도·규범 후보 탐색용으로만 쓴다.
    - (b) **조번호 필수.** 인용은 원문에 실재하는 "제N조"/"제N조제M항" 형식으로만 쓴다(서술형 라벨 금지 — 조번호가 있어야 조문 원문 팝업이 붙는다).
    - (c) **인증키는 환경변수로만.** `LAW_OC` 값을 코드·프롬프트·파일·JSON·커밋 어디에도 적지 않는다(`.mcp.json`은 `${LAW_OC}` 참조만).
    - (d) **현행성 확인.** 조회 결과의 시행일·"시행 예정"·연혁 경고를 확인하고, 연혁·구법 서술을 현행으로 쓰지 않는다(규칙 10과 결합).

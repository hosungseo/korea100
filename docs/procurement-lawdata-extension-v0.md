# 법제처 확장 데이터 연계 계획 v0

기준일: 2026-07-12

국가법령정보 공동활용 Open API는 법령·행정규칙 외에 판례, 헌법재판소 결정례, 행정심판례, 위원회 결정문(11개 위원회), 중앙부처 법령해석(35개 이상 부처) 등 약 191종을 같은 인증값(`LAW_OC`)으로 제공한다(2026-07-12 open.law.go.kr 가이드 확인). 이 문서는 조달 버전에서 어떤 데이터를 어떻게 연계할지 정한다.

## 1. 왜 연계하나

법령 조문은 제도의 설계도이고, 해석례·판례·심판례는 그 제도가 **실제로 다퉈진 기록**이다. 세 이용자 모두에게 조문만으로 답이 안 나오는 질문의 답이 여기에 있다.

- 조달업체: 부정당제재 취소소송, 입찰무효 다툼, 보증금 귀속 분쟁의 실제 결말
- 수요기관: 조달요청 의무 범위, 수의계약 사유 적용에 대한 유권해석
- 계약공무원: 기재부·행안부·법제처 해석으로 재량 판단의 근거 확보

## 2. 데이터 유형별 용도와 우선순위

| 순위 | 데이터 | 조달 도메인 용도 | 연계 대상 제도 예 |
| --- | --- | --- | --- |
| 1 | 법령해석례(법제처)·부처별 법령해석 | 수의계약 사유, 계약금액 조정 요건, 제재 요건 등 실무 쟁점의 유권해석 | 수의계약, 계약방법 결정, 물가변동 조정 |
| 2 | 판례(대법원 등) | 부정당제재 취소, 입찰무효, 계약금액조정 청구, 담합 손해배상 | 부정당업자 제재, 입찰 절차 전반 |
| 3 | 행정심판례 | 제재처분·등록말소 등 처분 불복의 재결 경향 | 부정당업자 제재, 조달업체 등록 |
| 4 | 위원회 결정문 | 공정거래위원회 입찰담합 의결 등 | 입찰 담합 대응 |
| 5 | 헌법재판소 결정례 | 제재 제도의 위헌성 판단 등 소수 쟁점 | 부정당업자 제재 |

법령용어·조문 관계 데이터는 검색 개선용 후보로만 기록하고 v0 범위에서 제외한다.

## 3. 오류 방지 원칙의 적용

확장 데이터는 법령보다 환각·오독 위험이 크다. 다음을 지킨다.

1. **자동 표시는 메타데이터만**: 사건·안건번호, 제목, 회신·선고 일자, 기관, 공식 원문 링크. 이 값들은 API 응답에서 그대로 가져오며 가공하지 않는다.
2. **요약은 원문 대조 후에만**: 해석례·판례의 한 줄 요지를 싣려면 원문을 열람하고 원문 문구에 근거해 쓴다. 요지 작성 규칙은 원문 우선 워크플로와 동일하다.
3. **선별은 사람이 한다**: 검색어 기반 자동 연결은 관련성 오판을 낳는다. 제도별 연계 항목은 제작자가 원문을 읽고 선별하며, 선별 이유를 한 줄 기록한다.
4. **시점 경고**: 해석례·판례는 당시 법령을 전제한다. 각 항목에 원문 일자를 표시하고, 근거 조문이 그 후 개정된 경우 `개정 전 법령 기준` 표시를 검토한다.
5. **개수 절제**: 제도당 대표 3~5건. 판례 데이터베이스가 아니라 제도 이해를 돕는 각주다.

## 4. 스키마 확장 (데이터 계약 v3 후보)

```ts
relatedDecisions?: Array<{
  type: "interpretation" | "precedent" | "adjudication" | "committee" | "constitutional";
  issuer: string;        // "법제처", "대법원", "중앙행정심판위원회", "공정거래위원회" 등
  caseNo: string;        // 안건·사건번호 (API 원문 그대로)
  title: string;         // 제목 (API 원문 그대로)
  decidedOn: string;     // YYYY-MM-DD
  officialUrl: string;   // law.go.kr 원문 링크
  relevance: string;     // 이 제도와의 관련성 한 줄 (제작자 작성)
  gist?: string;         // 원문 대조를 거친 한 줄 요지 (선택)
}>
```

UI는 상세 페이지 하단에 `관련 해석례·판례` 섹션으로 배치하고, 항목 클릭 시 국가법령정보센터 원문으로 이동한다. `validate:data`에 officialUrl·일자 형식 검사를 추가한다.

## 5. 파이프라인

- `sync:sources` 계열에 확장 데이터 조회 스크립트를 추가한다(`LAW_OC` 동일). 검색은 스크립트가, 선별은 사람이, 연결·형식 검증은 다시 스크립트가 맡는다.
- 주간 freshness 점검 대상에는 포함하지 않는다. 해석례·판례는 원문이 변하지 않으므로, 대신 근거 조문의 개정 감지 시 관련 항목의 시점 경고를 재검토한다.

## 6. 단계

1. **지금**: 이 계획 확정. 스키마 확장은 데이터 계약 v3에 반영 대기.
2. **파일럿 10개 제작 중**: 쟁점이 많은 2~3개 제도(수의계약, 부정당업자 제재)에 시범 적용해 선별 기준과 작성 비용을 측정한다.
3. **파일럿 공개 후**: 측정 결과로 전 제도 확대 여부와 제도당 항목 수 기준을 정한다.

전제: `LAW_OC` 확보. 법령 기계 대조와 같은 차단 요인을 공유한다.
